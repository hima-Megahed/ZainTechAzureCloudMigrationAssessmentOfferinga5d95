In scope

What the engagement covers for the selected application and supporting components.

Application and platform coverage

  • Application types: Web apps, APIs, background workers, frontends
  • Target platforms: Azure App Service, Azure Container Instances (ACI)
  • Database: Small dataset migration to Azure SQL Database or MySQL Flexible Server
  • Outputs: Target architecture diagram and modernization assessment report

Supported stacks

  • .NET Core / .NET MVC
  • Java (Spring Boot)
  • Python (Flask / Django)
  • PHP (Laravel)
  • Node.js (Express)
  • React / Angular / Vue.js

Containerization is supported for modern runtimes: .NET Core, Java, Python, Node.js, PHP.

Out of scope / exclusions

Items not included in this offering (can be planned as a follow-on phase).

Modernization exclusions

  • Multi-application or multi-tenant work (only one selected app)
  • Non-web workloads (desktop, mainframe, thick clients)
  • Complex refactoring or multi-service decomposition
  • Performance optimization beyond basic validation

Security and operations exclusions

  • Integration with on-prem authentication (ADFS/SAML/hybrid join)
  • Security hardening, DR setup, or managed services
  • Full dependency mapping
  • Third-party license remediation
  • Advanced DevOps automation or detailed pipeline design

Assumptions

Constraints used to keep the engagement fast, focused, and low-friction.

Application sizing

  • 1 small-to-medium web/API app (up to 3 logical components)
  • Up to 500 MB including dependencies
  • Up to 2 external APIs/integrations
  • Up to 10 GB total data (migratable over VPN)

Access and readiness

  • Client provides source code access and required credentials
  • Client provides Azure subscription for deployment and testing
  • VPN or ExpressRoute available if DB migration is needed
  • Existing code builds successfully in current environment
  • Dependencies are documented and customer has rights to modify source

Roles and responsibilities

A simple split to ensure speed and clarity during delivery.

ZainTECH

  • Modernization assessment and target design
  • Build packaging and deployment to App Service or ACI

  • Basic validation and knowledge transfer
  • Final report and architecture diagram

Customer

  • Provide Azure subscription and approvals for deployments

Engagement timeline

A lightweight plan designed for speed without sacrificing quality.

1
Kickoff and discovery (Days 1-2) Confirm scope, validate access, collect code and environment details, and identify quick wins.
2
Modernization plan and design (Days 3-4) Choose target platform (App Service or ACI), define target architecture, and confirm deployment approach.
3
Build and deploy (Days 5-8) Package the application, configure Azure resources, deploy, and optionally implement a starter CI/CD pipeline.
4
Validate and handover (Days 9-10) Basic testing, walkthrough session, and finalization of report and architecture diagram.
✓
Total effort Approximately 10 man-days (may vary based on access readiness and application complexity).

Key risks and dependencies

Typical factors that can impact timeline or feasibility for a quick modernization.

Access and approvals

Delays in repo access, secrets, networking approvals, or Azure permissions can extend delivery.

Hidden dependencies

Undocumented integrations, legacy libraries, or on-prem constraints may require additional remediation.

Data constraints

Data volume above the assumed limits or strict connectivity controls can shift the approach and duration.

Security requirements

Advanced security hardening, private endpoints, or compliance requirements may be handled as a follow-on phase.
